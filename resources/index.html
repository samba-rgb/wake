WAKE

Advanced Kubernetes Log Analysis & Diagnostics

Wake is a powerful, multi-pod Kubernetes log analysis and diagnostics platform. Beyond traditional log tailing and grepping, Wake enables real-time log filtering with advanced pattern matching, interactive terminal UI (TUI), web-based viewing with OpenObserve integration, template-based diagnostics (JFR, heap dumps, thread dumps), custom script execution across pods, and intelligent command history powered by TF-IDF search. All with seamless multi-pod support, intelligent sampling, and multi-threaded processing for massive clusters.

Web View (with OpenObserve)

Wake can send filtered logs to a web endpoint for viewing. This is great for sharing and collaboration. It integrates with OpenObserve.

Step 1: Start OpenObserve with Docker

bash 
# Create data directory for OpenObserve
mkdir -p data
chmod +x data

# Start OpenObserve container
docker run -d --name openobserve \
  -p 5080:5080 \
  -e ZO_ROOT_USER_EMAIL="root@example.com" \
  -e ZO_ROOT_USER_PASSWORD="Complexpass#123" \
  -v $(pwd)/data:/data \
  public.ecr.aws/zinclabs/openobserve:latest

What this does:

  • mkdir -p data ‚Äì Create data directory for persistent storage
  • chmod +x data ‚Äì Make directory executable
  • -d ‚Äì Run in background
  • --name openobserve ‚Äì Container name for easy management
  • -p 5080:5080 ‚Äì Expose port 5080 for web UI
  • -v $(pwd)/data:/data ‚Äì Mount local data directory for persistence
  • -e ZO_ROOT_USER_EMAIL ‚Äì Set login email
  • -e ZO_ROOT_USER_PASSWORD ‚Äì Set login password

Step 2: Verify OpenObserve is Running

bash 
# Check container status
docker ps | grep openobserve

# View logs
docker logs -f openobserve

Once running, access OpenObserve at: http://localhost:5080

Step 3: Configure Wake to use OpenObserve

bash 
wake setconfig web.endpoint "http://localhost:5080"
wake setconfig web.user "root@example.com"
wake setconfig web.password "Complexpass#123"

Step 4: Send Logs to Web View

bash 
# Send all logs from "my-app" pod containing "error"
wake "my-app" -i "error" --web

# Send all logs from all pods in production namespace
wake -n production -A --web

# Send logs and save to file simultaneously
wake "my-app" --web -w /tmp/logs.txt

üìå Note: Stream names are auto-generated daily (e.g., logs_wake_{namespace}_2025_11_30). Logs appear in OpenObserve UI immediately.

Advanced Log Filtering

Wake supports advanced filtering to help you find exactly what you need.

bash 
# Include logs that contain "error" OR "warn"
wake "my-app" -i '"error" || "warn"'

# Exclude logs that contain "debug"
wake "my-app" -e "debug"

# Complex query: (error OR warn)
wake "my-app" -i '"error" || "warn"'

Kubernetes Context Management

Wake automatically uses your current Kubernetes context. The context is detected from your kubeconfig, so you don't need to specify it explicitly in most cases. No namespace tensions or conflicts!

‚ö†Ô∏è Important: Wake uses your default Kubernetes context to connect to your cluster. Make sure your kubeconfig is configured correctly with the right context set. If you're unsure which context you're using, run kubectl config current-context. You can also explicitly specify a namespace with -n <namespace> or use -A for all namespaces.

How Context Detection Works

Wake uses the context configured in your kubeconfig file. By default, it uses the current context set in your cluster. The context you're in determines which cluster Wake connects to.

Check Your Current Context:

bash 
# View current context
kubectl config current-context

# List all available contexts
kubectl config get-contexts

Switch Kubernetes Context with kubectx (Recommended)

kubectx is a fast, interactive tool for switching between Kubernetes contexts. It's much faster and easier than kubectl:

Install kubectx:

bash 
# macOS with Homebrew
brew install kubectx

# Or download from GitHub
# https://github.com/ahmetb/kubectx

Use kubectx to Switch Contexts:

bash 
# List all contexts (interactive)
kubectx

# Switch to a specific context
kubectx production-cluster

# Switch to previous context (like 'cd -')
kubectx -

Wake Automatically Uses Your Context

Once you've switched contexts using kubectl or kubectx, Wake will automatically use that context for all commands. No configuration needed!

Example Workflow:

bash 
# 1. Switch context to production
kubectx production-cluster

# 2. Wake automatically uses this context
wake "my-app" -i "error"

# 3. Switch to staging
kubectx staging-cluster

# 4. Wake now uses staging context
wake "my-app" -i "error"  # Same command, different cluster!

Override Context (Advanced)

If needed, explicitly specify a context instead of using the current one:

bash 
# Use a specific context
wake -x production-cluster "my-app" -i "error"

# Use custom kubeconfig file
wake -k ~/.kube/custom-config "my-app"

Context + Namespace (No Conflicts)

Context and namespace work together seamlessly. No namespace tensions:

bash 
# Logs from 'default' namespace in current context
wake "my-app"

# Logs from 'production' namespace in current context
wake -n production "my-app"

# Logs from all namespaces in current context
wake -A "my-app"

# Switch context and get logs from specific namespace
kubectx staging && wake -n payment-service "api-pod"

Templates

Templates enable you to execute predefined diagnostic commands inside your pods with support for multiple execution modes. Run diagnostics like Java Flight Recorder (JFR), heap dumps, and thread dumps efficiently across your entire cluster.

Template Types

Wake supports multiple diagnostic templates:

  • JFR (Java Flight Recorder) - Profile Java applications with low overhead
  • Heap Dump - Capture heap snapshots for memory analysis
  • Thread Dump - Analyze thread states and stack traces

List Available Templates

bash 
wake --list-templates

Execution Modes

Templates support flexible execution across different pod selection strategies:

Execute on Single Pod

bash 
# Execute thread-dump on a specific pod
wake "my-app" --exec-template thread-dump --template-args 1234

Execute on All Matching Pods (Multi-threaded)

bash 
# Execute heap-dump on all pods matching "backend" in all namespaces
wake -A "backend" --exec-template heap-dump --template-args 1234

Execute with Pod Selector (Regex)

bash 
# Execute JFR on pods matching a regex pattern in namespace
wake -n production "^api-service-.*" --exec-template jfr --template-args "30s"

Execute with Sampling (Random Subset)

bash 
# Execute thread-dump on a random sample of 5 pods from matching selection
wake -A "worker" -s 5 --exec-template thread-dump --template-args 1234

Scripts

Execute custom scripts inside your pods with flexible pod selection and multi-threaded parallel execution. Perfect for running diagnostics, health checks, or maintenance tasks across your cluster.

Execution Modes

Scripts support multiple execution strategies for different use cases:

Execute on Single Pod

bash 
# Run a script on a specific pod
wake "my-app" --script-in ./scripts/check_health.sh --script-outdir /tmp/results

Execute on All Matching Pods (Multi-threaded)

bash 
# Run script on all pods matching a pattern across all namespaces
wake -A "worker" --script-in ./scripts/check_disk.sh --script-outdir /tmp/disk_usage

Execute with Pod Selector (Regex)

bash 
# Run script on pods matching regex in specific namespace
wake -n production "^cache-.*" --script-in ./scripts/cache_stats.sh --script-outdir /tmp/cache

Execute with Sampling (Random Subset)

bash 
# Run script on a random sample of 10 pods (useful for large clusters)
wake -A "service" -s 10 --script-in ./scripts/performance_check.sh --script-outdir /tmp/perf

Command History (TF-IDF Search)

Wake saves your command history and allows you to search it using TF-IDF for intelligent retrieval.

bash 
# Search for commands related to "error logs"
wake --his "error logs"

# Show all command history
wake --his

Example: wake --his jfr

Response:

Useful args: -c <container>, -n <namespace>, -s <sample>, -i <include>, -e <exclude>.

Update Manager

Keep Wake up to date with the latest features and improvements.

Versioning

Wake follows semantic versioning: MAJOR.MINOR.PATCH.

  • MAJOR ‚Äî incompatible API or behaviour changes (breaking changes).
  • MINOR ‚Äî new features, backwards-compatible.
  • PATCH ‚Äî bug fixes and small improvements, backwards-compatible.

Examples: 1.2.0 ‚Üí new features since 1.1.x; 2.0.0 ‚Üí breaking changes from 1.x.

Check the Wake version locally:

bash 
wake --version
# or
wake -V

Update via Homebrew (macOS)

Update the Homebrew package on macOS:

bash 
brew upgrade samba-rgb/wake/wake
